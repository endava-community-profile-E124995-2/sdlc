# Repository Guidance

- Use the `$product-requirements-manager` skill for product discovery, PRD drafting or refinement, acceptance criteria, prioritization, launch readiness, metric design, and compliance-sensitive requirement mapping.
- Use the `$architecture-and-engineering-manager` skill for technical feasibility, architecture ownership, interface and integration design, engineering execution planning, dependency sequencing, and cross-role engineering handoffs.
- Use the `$quality-and-security-and-governance-manager` skill for independent quality review, security boundary review, control mapping, evidence sufficiency review, exception framing, and release-gate recommendations.
- Use the `$deployment-and-operations-manager` skill for release planning, rollout sequencing, rollback readiness, observability and runbook framing, support handoff, and post-release operational follow-up.
- Do not use the product skill as the owner of architecture, implementation design, generic coding or debugging, or approval decisions for legal, compliance, finance, or security.
- Do not use the deployment skill as the owner of product discovery, architecture design, implementation planning, or release-gate approval decisions.
- If a request mixes product framing with engineering execution, use the product skill for problem framing, scope, rules, and acceptance criteria, then use the architecture skill for technical design, implementation planning, and engineering handoffs.
- If a request mixes design or delivery work with approvals, evidence gaps, control interpretation, or release-gate questions, keep product or architecture as the owner of their domain and hand the assurance, control, and gate work to the quality skill.
- If a request mixes launch framing with runtime rollout execution, use the product skill for audience, enablement, and business rollout constraints, then use the deployment skill for rollout sequence, rollback, observability, runbooks, and support ownership.
- If a request mixes deployment planning with unresolved runtime boundaries, keep deployment as the owner of operational consequences and hand the design decision to the architecture skill.
